# Context Awareness - Operating Guide (Reference)

## Design goals

- **Deterministic discovery**: `docs/context/AGENTS.md` is the authoritative LLM routing entrypoint with a fixed progressive loading protocol.
- **Stable entry point**: `docs/context/INDEX.md` and `docs/context/registry.json` are the supported context entry points.
- **Semantic correctness**: OpenAPI quality gate blocks incomplete endpoint definitions before merge.
- **Verifiable updates**: artifact checksums enable CI to detect edits that bypass the scripts.
- **Knowledge access**: domain glossary and architecture principles available as structured artifacts.
- **Tool-agnostic artifacts**: OpenAPI, BPMN 2.0, and a normalized DB schema mapping.

## Contract mode vs Generated mode

### Contract mode (recommended default)

- The artifact file is the authoritative contract.
- Human collaborators and LLMs edit the file directly.
- After edits, run:
  - `node .ai/skills/features/context-awareness/scripts/ctl-context.mjs touch`
  - `node .ai/skills/features/context-awareness/scripts/ctl-context.mjs verify --strict`

### Generated mode (opt-in)

- The artifact file is generated from external tools (e.g., OpenAPI generators, schema extractors).
- Register the artifact with `mode=generated` and optionally document the source command in `source.command` for reference.
- **ctl-context does NOT execute generators automatically**. Run your generator manually, then:
  - `node .ai/skills/features/context-awareness/scripts/ctl-context.mjs touch`
  - `node .ai/skills/features/context-awareness/scripts/ctl-context.mjs verify --strict`
- The `source.command` field is metadata for humans/LLMs to know how to regenerate; it is not executed by ctl-context.

## OpenAPI quality gate

Before committing OpenAPI changes, validate semantic completeness:

```bash
node .ai/scripts/ctl-openapi-quality.mjs verify --source modules/<id>/interact/openapi.yaml --strict
node .ai/scripts/ctl-openapi-quality.mjs verify --discover-modules --strict
```

The pre-commit hook runs this automatically on staged OpenAPI files.

## Recommended CI gates

Minimum:

- `node .ai/skills/features/context-awareness/scripts/ctl-context.mjs verify --strict`
- `node .ai/scripts/ctl-project-state.mjs verify`

Full api-context suite (run by `ci-verify.mjs` when OpenAPI files exist):

1. `node .ai/scripts/ctl-openapi-quality.mjs verify --discover-modules --strict`
2. `node .ai/scripts/ctl-api-index.mjs generate --touch`
3. `node .ai/scripts/ctl-api-index.mjs verify --strict`
4. `node .ai/skills/features/context-awareness/scripts/ctl-context.mjs verify --strict`
5. `git diff --exit-code` on generated artifacts (drift detection)

## Common artifact types

- `openapi`: API surface contract (per-module: `modules/<module_id>/interact/openapi.yaml`; project-level: `docs/context/api/openapi.yaml`)
- `api-index`: LLM-optimized API summary (`docs/context/api/api-index.json`; generated by `ctl-api-index.mjs`)
- `db-schema`: normalized DB structure mapping (`docs/context/db/schema.json`)
- `bpmn`: BPMN 2.0 process file (`docs/context/process/*.bpmn`)
- `glossary`: domain term definitions (`docs/context/knowledge/glossary.json`; managed by `ctl-context add-term/remove-term`)
- `architecture-principles`: cross-cutting constraints (`docs/context/knowledge/architecture-principles.md`)

## Troubleshooting

- **Checksum mismatch**: you edited an artifact but did not run `ctl-context touch`.
- **Missing file**: registry references a path that does not exist; create the file or remove the registry entry via script.
